## Deferred / Nice-to-have

- [ ] **Portal-side licensing gate** ‚Äî once shipped, drop `DEV_SKIP_LICENSE` in `services/robot/`. Today the robot `/api/robot/*` endpoints are gated by a dev-bypass env var; the Portal is the intended source of truth for per-serial licensing.

- [ ] **Relfar DB tables + endpoints** ‚Äî Relfar controller UI is live under `services/relfar/` but Portal has no tables for laser sessions / telemetry yet. Device sync should mirror the robot pattern (serial-keyed, snapshot + audit).
- [ ] **"Replace machine" super-admin flow + memory transfer** ‚Äî when a client swaps a broken/retired unit, a super-admin should be able to mark the old Machine retired and bind a new serial to the same Station slot while copying the "robot memory" (pathways, obstacles, safety zones, self-awareness data) over. Today the station-detail card already locks serial + taxonomy once a Machine row exists; the missing pieces are: (a) a `robotMemory` JSONB column on `Machine` (or a sibling `MachineMemory` table if blobs get large), (b) `POST /api/machines/:id/replace { newSerial, copyMemory: true }` that creates the new row, copies the blob, retires the old row (status=`decommissioned`), and rewires `notes.machineData[i].machineId`, (c) role gate ‚Äî only super-admins see the "Replace" button on the locked card.
- [ ] **Refactor admin/machines list page for the new taxonomy** ‚Äî `src/app/admin/machines/page.tsx` still thinks in `type: 'robot' | 'laser'` (filter chips, create form, icons). It keeps working today via the derived `type` alias returned by `/api/machines`, but the list can't surface the real category/subcategory/model triple or the new models (E05/E10/E12 variants, Traditional welding, Sanding). Rework the filter UI into Category + Subcategory + Model and swap the create form over to the new fields. Low urgency ‚Äî the Stations tab is now the primary machine-creation path.
- [ ] **Monitor-only laser sources ‚Äî handle in Relfar, stream telemetry up to Portal UI (decided 2026-04-15)** ‚Äî Hugo's preferred path is to keep the Portal data model simple: don't model the external laser source as a separate `Machine` row. Instead, detect/monitor it inside `services/relfar/` (one source present or not) and forward telemetry to the Portal UI over the existing device-sync channel. Portal just renders the telemetry as part of the Relfar machine's card ‚Äî no new table, no new MAC/serial plumbing. Revisit ONLY if we later decide the source needs to be a first-class Portal entity (e.g. for per-source licensing, warranty tracking, or swapping sources between controllers). The deferred `MachineInterface` / multi-MAC refactor is parked until that day ‚Äî current `Machine.macAddress` singular is fine for everything we actually ship today. Hardware-swap flow is still covered by the planned super-admin "Replace machine" memory-transfer.
- [ ] **Thin-client migration** ‚Äî long-term IP-protection goal. Move station-PC logic + UI into Portal so the PC becomes a thin executor. Tracked separately; any new feature should prefer the portal side when there's a choice.

- [x] **Station PC ‚Üî Station assignment audit trail + return-to-pending on station delete** *(2026-04-15)* ‚Äî new `StationPCAssignment` table (migration `20260415_station_pc_assignment_history.sql`) records every assign/detach event with actor (best-effort from `auth-token` cookie), reason (`manual` / `reassigned` / `station_deleted` / `pc_retired`), station snapshot (id + number + title ‚Äî snapshot survives Station deletion via no-FK + stored columns). Writes happen in three places: (a) `DELETE /api/stations/[id]` now detaches any bound PC, flips it back to `provisioning` + `approved=false` (unless already retired), and logs `action='detached' reason='station_deleted'` ‚Äî exactly the behaviour Hugo asked for; (b) `PATCH /api/station-pcs/[id]` logs a `detached` event for each prior Station the PC was bound to (reason = `reassigned` if moving to a new station in the same call, `manual` otherwise) plus an `assigned` event for the new target; (c) when a target Station already had a different PC bound, that displaced PC also gets a `detached` audit row and is sent back to pending. New `GET /api/station-pcs/[id]/history` endpoint, and a small collapsible history block at the bottom of the Station PC detail panel (shows last 3 events by default, expandable). Helper: `src/lib/actor.ts#getActorFromRequest` decodes the auth cookie without throwing ‚Äî audit rows still get written when the token is missing, just with `actorEmail=null`.
- [x] **Fix: invoice-removal cleanup is now server-authoritative + heals orphan items** *(2026-04-15)* ‚Äî Hugo hit the ghost-machines bug again on a station that should have had 2 machines but showed 3, and reported that removing one invoice "removed all of them" (the single remaining chip row just disappeared because the UI only rendered chips when `linkedInvoices.length > 1`). Two fixes: (1) `DELETE /api/stations/[id]/invoices/[invoiceId]` is now authoritative ‚Äî it deletes the StationInvoice row AND reconciles `notes.items` / `notes.machineData` / `notes.invoices` / primary `invoiceNumber` in a single transaction. Items are dropped if they (a) are tagged with the removed invoice, (b) carry a tag pointing at any invoice that no longer has a surviving DB row (self-heal for orphans left by prior buggy removals), or (c) are untagged AND the removed invoice was primary (untagged items are implicitly primary-owned). `machineData` is re-indexed to match the kept items so later "add invoice" cycles don't pile cards onto orphan slots. Primary promotion now prefers a surviving DB StationInvoice over the legacy `notes.invoices` array. Frontend `handleRemoveStationInvoice` dropped its brittle two-step optimistic PATCH for the DB path ‚Äî just DELETE + `refreshStations()`. (2) Changed the linked-invoices chip visibility from `length > 1` to `length > 0` so the remaining chip stays visible after a removal ‚Äî makes it obvious that only the clicked invoice was unlinked.
- [x] **Fix: assigning a retired Station PC now revives it** *(2026-04-15)* ‚Äî in `/api/station-pcs/[id]` PATCH, the implicit-approval branch for `assignToStationId` was explicitly *skipping* retired PCs (`existing.status !== 'retired'`). Side-effect: once a PC was retired, attaching it to a new station left its badge stuck on "Retir√©" forever (Hugo hit this with BN1504BF70842 attached to STN-001). New behaviour: assigning a PC to a station is treated as a deliberate revival ‚Äî if the PC was retired and the caller didn't explicitly pass `status`, flip `status: 'retired' ‚Üí 'provisioning'` (actual `online` state will follow from the next heartbeat). Also flips `approved: true` on revival unless caller overrides. Unapproved-but-not-retired PCs keep the existing auto-approve behaviour. Explicit `approved: false` / explicit `status: ...` overrides are still honoured.

- [x] **Fix: orphaned line items after invoice removal ‚Üí ghost machines** *(2026-04-15)* ‚Äî removing an invoice from a station via the Clients tab left that invoice's line items in `notes.items`, so the next "add invoice" stacked a new item on top and the station showed N+1 machines (Hugo saw 3 on station 1486 after a 2‚Üí1‚Üí2 invoice cycle).
